Skriv ditt svar här:

Jag genomförde LIA 2 i ett utvecklingsprojekt där jag byggde en fullstack IoT-lösning för övervakning av miljösensordata (badvattenkvalitet). Projektet kan liknas vid en verksamhet som utvecklar en digital tjänst för att samla in, lagra och visualisera mätdata samt hantera avvikelser.

Vad gör verksamheten?
- Utvecklar en plattform som tar emot sensordata via ett API, lagrar den som tidsserier och gör den tillgänglig i en dashboard.
- Tillhandahåller larm (alert rules) och notifieringar (e-post) när data indikerar avvikelse.
- Stödjer ett ärendeflöde där användare kan rapportera incidenter och där roller styr åtkomst.

Vilka kunder och tjänster?
- Tjänsten riktar sig till verksamheter som behöver uppföljning av miljödata över tid, t.ex. kommuner, driftansvariga eller analysfunktioner.
- Kärntjänsten är datainsamling + visualisering (status och trender), kompletterat med larm och ärendehantering.

Vad kännetecknar arbetssättet?
- End-to-end dataflöde (sensor → datakommunikation → dataplattform → presentation).
- Fokus på datakvalitet och spårbarhet: validering av indata, historik och loggning av larmhändelser.

Varför valde jag detta?
Jag valde projektet för att få träna hela kedjan från IoT-arkitektur och datakommunikation till dataplattform/datalagring och integration med externa tjänster (SMTP). Det passade även mitt mål att arbeta mer metodiskt och leverera en fungerande helhet.

Mina personliga mål med LIA 2:
- Bygga en stabil IoT-kedja (sensor → API → databas → dashboard) med tydliga API-kontrakt.
- Fördjupa mig i dataplattform och dataintegration (validering, historik och åtkomst via API).
- Implementera larmregler och e-postnotifieringar som fungerar i praktiken.
- Utveckla arbetssätt: planering, prioritering, kvalitetssäkring och felsökning i ett fullstackflöde.

Skriv ditt svar här:

Jag arbetade “över flera delar av systemet”:
- Sensor/IoT-del: Python-sensorsimulator.
- Backend/API: Django + Django REST Framework (DRF).
- Dataplattform: SQLite via Django ORM.
- Frontend/dashboard: HTML + Bootstrap + Chart.js.
- Notifieringar: e-post via Djangos mail-stöd (console backend i utveckling eller SMTP i “skarpare” läge).

Mina arbetsuppgifter:
- Utveckla sensorsimulatorn som läser ett öppet dataset (CSV), skapar tidsstämplade mätningar och skickar JSON via HTTP POST till backend (inkl. konfigurerbart intervall och retry vid nätverksfel).
- Bygga REST-API för att ta emot data, validera payload (serializers), spara i databas och exponera senaste värde och historik till dashboarden.
- Skapa/vidareutveckla datamodeller och migreringar för sensordata, larmregler (AlertRule), larmhändelser (AlertEvent) samt ärenden (IncidentReport).
- Implementera larmregler som utvärderar inkommande sensordata och skickar e-post när villkor uppfylls, samt loggar händelser och använder cooldown för att undvika spam.
- Bygga dashboard-vyer och templates som visualiserar data (aktuella värden och historik) och stödjer inloggning/roller samt formulär för ärenden.
- Testa och felsöka integrationen mellan sensor ↔ API ↔ databas ↔ dashboard samt e-post via SMTP.

Relatering till klassrumsbaserade kurser (hur jag tillämpade dem):
- Dataplattformar: relationsdatabas (SQLite) + ORM för schema, historik och filtrering.
- Dataintegration: API-kontrakt, validering och endpoints för ingestion (POST) och konsumtion (GET latest/history/export).
- Molnlösningar: integration med extern e-posttjänst via SMTP och konfiguration via miljövariabler.
- IoT-arkitektur och datakommunikation: HTTP/REST, tidsstämplar/tidszoner och robusthet (retry/felhantering).
- Projektmetodik: iterativt arbetssätt med små leveranser och fokus på fungerande end-to-end innan fler funktioner.

Skriv ditt svar här:

Jag fick tydliga erfarenheter av att IoT-system fungerar först när hela kedjan fungerar, inte bara enskilda komponenter. Många problem uppstod i gränssnitten mellan delarna (payload, datatyper, tidszon, behörigheter eller konfiguration).

Något som hände under perioden som var viktigt att tänka närmare på:
E-postnotifieringar via SMTP visade sig vara mer än “bara kod”. Jag behövde förstå leverantörens krav (t.ex. verifierad avsändardomän), rätt inställningar (host/port/TLS) och hur man felsöker när utskick nekas eller inte levereras.

Varför blev det jag gjorde bra eller mindre bra?
- Bra: jag började med en minimal end-to-end-version (sensor → API → databas) och verifierade att data flödade korrekt innan jag byggde vidare med dashboard, larm och ärenden.
- Mindre bra: när jag förändrade flera delar samtidigt blev felsökning svårare. Det lärde mig att avgränsa förändringar och testa i små steg.

Hur gjorde jag det och vilka konsekvenser fick det?
Jag jobbade iterativt: byggde en del, testade lokalt (simulator + API-respons + databas + UI), integrerade och justerade. Konsekvensen blev att jag kunde koppla fel till rätt lager (kod vs konfiguration) och få stabilare leveranser.

Hade jag kunnat lösa uppgiften på ett annat sätt?
Ja. För en mer produktionslik lösning hade jag kunnat använda MQTT/broker eller kö-system för ingestion och en produktionsdatabas (PostgreSQL) från början. För projektets scope var REST + SQLite rimligt, men jag ser tydligt hur man skulle vidareutveckla lösningen.
